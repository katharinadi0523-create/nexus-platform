# Nexus Platform 开发规范

## 1. 开发原则

- 修改前先读当前代码、搜索已有实现并检查 `git status`。
- 保持最小、聚焦的 diff，不修改或覆盖无关内容。
- 先确认配置态、定义态和运行态的真实关系，再实现界面。
- 未接入的能力不得表述为已经可用。

## 2. 优先复用组件

按以下顺序选择组件：
1. `components/ui/` 基础组件；
2. 当前业务域共享组件；
3. Runtime / 对话调试组件；
4. 确认无法扩展后再新建组件。
Claw 对话与调试优先复用 `conversation-timeline`、`debug-chat-composer`、`interactive-chat-panel` 和现有资源配置 Dialog。禁止复制一套相似实现；主 Claw 与子智能体使用同一能力模型。

## 3. 数据与功能解耦

- Mock/API 数据、领域规则、页面状态和 UI 渲染必须分层。
- Mock 放在 `lib/mock/` 或模块数据文件；展示组件通过 props 接收数据。
- 合并、筛选、启停和映射等规则写成纯函数。
- 使用稳定 `id` 建立关系，不用名称或数组下标充当外键。
- 通用组件不得硬编码具体命题、Agent 名称或工具映射。

## 4. Runtime 规则

- 配置资源必须与实际 Debug / Runtime 映射一致。
- 主对话、子会话和工具区共用同一份实例数据。

## 5. 页面与验收

- 分栏布局响应容器宽度；表格过窄时内部滚动，不挤压文字。
- 必须处理空态、错误态、禁用态、长文本和重复提交。
- 一个页面只突出一个主要动作，保持克制的企业级视觉。
- 完成前执行：

```ba
npx eslint <修改的 TypeScript 文件>
git diff --check
npm run build
```

- 关键交互必须在真实浏览器中验证。

## 6. 发布

- “发布并推送”默认包含当前工作区全部改动，包括github推送和线上部署两个动作。
- 默认直接发到main，除非用户指定分支。
- 上线后确认 Vercel 为 `READY`、已绑定 `https://nexus-af.rowandi.top`，并验证域名和受保护路由。
